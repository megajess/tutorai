# Ticket 004 — Chat API Endpoint

## Status
`Done`

## Type
`Feature`

## Summary
Wire the retrieval layer and Ollama LLM together behind a `POST /chat` handler in Go. This ticket connects all the backend pieces from ticket 003 into a working API that accepts a user query and returns a grounded natural language response.

## Acceptance Criteria
- [x] `POST /chat` accepts `{"query": "string"}` and returns `{"response": "string"}`
- [x] `backend/internal/llm/ollama.go` wraps `POST http://localhost:11434/api/chat` and returns the response text as a Go string
- [x] Handler runs intent classification, triggers the appropriate retrieval call(s), assembles context, calls Ollama, and returns the result
- [x] Returns a JSON error response (not a 500 panic) if Ollama is unreachable: `{"error": "LLM unavailable"}` with status 503
- [x] Returns a JSON error response if the data service is unreachable: `{"error": "data service unavailable"}` with status 503
- [x] Returns a non-empty response string if retrieval finds no relevant context (Ollama is called with a fallback prompt acknowledging limited context)
- [x] `GET /health` returns `{"status": "ok"}` with status 200
- [x] CORS middleware allows requests from `http://localhost:5173`
- [x] Integration test using `httptest` hits `POST /chat` with a sample query and confirms a non-empty `response` field is returned (mock the data service and Ollama with `httptest.NewServer`)

## Implementation Notes
- Ollama request body: `{"model": "<OLLAMA_LLM_MODEL>", "messages": [...], "stream": false}`
- System prompt should instruct the model to answer only from provided context and say so if context is insufficient
- Use `net/http` with a shared `http.Client` (with timeout) for both Ollama and data service calls — set in `config.go`
- CORS: use Chi middleware or a small inline middleware — allow `Content-Type` header and `POST` method from the frontend origin
- All config (Ollama URL, model name, data service URL/key) comes from `config/config.go`, loaded via `godotenv`
